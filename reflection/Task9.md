# Рефлексия

Выделение отдельного класса `Any`, помимо `General`, как по мне, очень удачное решение. Но есть вопросы к идее 
реализации "базовых" методов для всех классов сразу в `General`. Очевидно, что реализация большей части этих методов
будет завязана на рефлексии, что, как мне кажется, негативно скажется на их производительности. Но даже для эффективной
реализации этих методов потребуется либо использовать нативный код, либо использовать специальные библиотеки, что 
добавит лишние зависимости, хотя мы даже не преступили ещё к реализации самой задачи.  Может быть мне такой подход 
кажется чуждым, потому что в ЯП, на котором я пишу, такого нет. Точнее как, в Java в классе `Object` есть реализация 
этих "базовых" методов, но эта реализация примитивна и в большинстве случаев переопределяется для каждого конкретного
наследника.

Ещё у меня есть вопрос, должен ли класс `Any` быть абстрактным, на который я не смог для себя однозначно ответить. Да,
и непонятно, должны ли присутствовать значения для приватных полей в результатах методов, например клонирования, 
копирования, сериализации.

Что касается примеров решения, то, сравнивая своё решение, с решением на Python, я понял, что забыл применить знания с
первого курса по ООАП, а именно разделение методов на запросы и команды и добавление пред и пост условий. Мне сложно 
оценить решение на Python, так как я плохо знаком с этим языком, но вот решение на Java мне не нравится:
- отсутствует реализация метода "проверка типа";
- понятно, что самому реализовывать парсер JSON это слишком для этого задания, но всё равно есть факт использования
сторонней библиотеки Jackson. А также есть вопросы к тому, для чего для каждого вызова методов 
сериализации/десериализации нужно создавать новый объект `ObjectMapper`;
- непонятно, для чего было параметризовывать методы, если они все должны работать с типом `General`;
- у меня есть вопросы к методу десериализации, а точнее к тому, что он статический. Мне кажется, что это неверный 
дизайн. Считаю, что этот метод должен быть, по сути, конструктором. Что касается моей реализации этого метода, то ошибка 
в том, что я забыл добавить ещё один аргумент для типа объекта-результата;
- есть вопросы к реализации методов `equals` и `toString`. Так как они в качестве реализации используют реализации
соответствующих методов из класса `Object`, то есть вопросы к их полезности. Так, при вызове метода `equals` будут 
сравниваться не объекты, а их ссылки. А при вызове `toString` будет печататься имя класса и хэш-код объекта, что совсем
не информативно.
